Kubernetes，並成為 Production 環境一部份，更可以與 Google Cloud 無縫整合。</li><li>swarm 在 v1.12 後推出了 swarmkit <sup><a href="http://note.drx.tw/2016/09/container-summit-2016-it-container.html#swarmkit">2</a></sup> 整體功能完整了不少，也更友善些。</li><li>DC/OS 在國外已有知名的企業在使用，有漂亮的圖形化介面和豐富的 marketplace。</li><li>Rancher 有著簡單易上手的介面、流程和 marketplace。</li></ol><br />至於要怎麼抉擇？專家建議我們可以先了解各家 container cluster 的觀念和想解決的問題，之後再依自己的屬性選擇最適合的解決方案 (solution)。<br /><br />Rancher 因敝公司已有在使用在此就不多談，倒是會想找時間玩玩看 Kubernetes 和 swarm。而 DC/OS 似乎較偏愛 CentOS？這對較偏愛 Debian 和 Ubuntu 的凍仁較不友善，暫且不太會關注它。<br /><br /><h3>容器監控 (Monitoring)</h3>容器監控可說是凍仁目前最關注的議題，雖曾使用過 <a href="https://www.nagios.org/" target="_blank" title="The Industry Standard In IT Infrastructure Monitoring">Nagios</a> 和 <a href="http://www.zabbix.com/" target="_blank" title="The Enterprise-Class Open Source Network Monitoring Solution">Zabbix</a>，卻一直對 <a href="https://prometheus.io/" target="_blank" title="Monitoring system &amp; time series database">Prometheus</a> 不得其門而入，但還是得到了不少寶貴的心得分享，期許自己能早日掌握這項新技術。<br /><ol><li>Prometheus 目前已有非常多的 Exporter 可使用 (除官方提供外還有不少第三方的可使用)，如 MySQL 就有 400 多個 metrics。</li><li>Prometheus 可以對 Nginx, MySQL 的服務進行監控。</li><li>可以藉由在 metrics 裡加上 Label 來群組化 metrics。</li><li>使用 Prometheus 時，一開始會不知道應該對什麼進行監控，這時會先把 Exporter 都先載進來，之後再慢慢觀察有什麼需要監控；與使用 Zabbix 時一開始就知道要監控什麼的觀念相反。</li><li>若已有 Zabbix server，可將收到 error，後送到 Pushgateway 丟給 Prometheus 處理。</li></ol><br /><center><table class="dark" style="text-align: center;"><tbody><tr>     <th></th>     <th>Zabbix</th>     <th>Prometheus</th>   </tr><tr> Manager</td>   </tr></tbody></table><br />▲ 不負責的 Zabbix 與 Prometheus 名詞對照表。<br /></center><br /><h3>資訊安全 (Security)</h3>Docker 安全嗎？Docker 官方的回應是 <b>Security by default</b>。在預設環境下他人是有機會從 container 裡黑到 host 的，很多問題牽扯到 Linux, cGroup, Namespace, Capability 等較底層的實作，雖然 Docker 不是為了 security 而生，但還是可以透過些手法來增強安全性。<br /><ol><li>限制 container 的 CPU, memory, ... 等使用資源。</li><li>TLS 加密很重要，否則世界會被 cracker 搶走！</li><li>啟用 SELinux 和 AppArmor 強化權限限制。</li><li>不要在 Docker container 裡使用 root 執行 service。</li><li>檔案最小權限設置，不該被寫入的、被執行的權限就該拔。</li><li>移除不必要的檔案或程式，例如 make, gcc ...。</li><li>量身打造 docker image。</li></ol><br />在 Capabilities 的 demo 中，講者 孫宏亮更透出可以在 container 裡控制 docker-engine (Docker Server) 的手法！<br /><blockquote><span class="Comment"># 在 container 裡控制 docker-engine (nonwithout TLS)，之後就可以控制全世界！</span><br /><pre>[ <span style="color: #ffdb00;">root</span><span style="color: red;">@docker-container</span> <span style="color: #ad7fa8;">~</span> ]<br />$ docker -H tcp://$IP:2375 run -it --pid host --ipc host --privileged -v /:/new_data ubuntu:14.04 bash </pre></blockquote><br />講者 徐千洋 (Tim) 則透過遠端命令執行 (Remote Code Execute) 的手法讓人了解可以藉由把 Docker 把服務打包成更安全的環境，以簡化營運成本。<br /><br /><h3>企業導入經驗分享</h3><div style="font-size: large;"><b>把 Docker container 跑起來了，然後呢？</b> </div><br />這是剛入門 Docker 時大家都會遇到的問題。凍仁一開始用它來 debug Vim Taiwan 社群伙伴的 vimrc，之後幫前公司架了私有的 <a href="https://docs.docker.com/registry/" target="_blank" title="類似 Git repository server，存放 Docker repository 用的。">Docker Registry</a>，也成功的在 Linux server 上跑了 PHP 7.0 給新專案使用 (同台機器上的舊專案還在 PHP 5.4 的環境)。<br /><br />自己用的很開心，但想全面導入卻困難重重。在高雄時身邊沒幾位朋友可以討論，雖有少數同事有興趣，但也因時程壓力下遲遲無法推動。在聽完 <a href="http://containersummit.ithome.com.tw/speaker.html#s1180" target="_blank">William Yeh</a> 前輩的 <a href="http://school.soft-arch.net/blog/268878/embrace-docker" target="_blank">Docker 導入：障礙與對策</a>的演講後，凍仁簡單列了幾個問題：<br /><ol><li>原有架構把 session、PDF 等資料存放在 server 本機上，不利 scale。</li><li>未導入自動化測試，無法保證更改架構後照成的影響。</li><li>尚未掌握在 Production 環境跑 Docker 技術，例如：Kubernetes (K8s), DC/OS (Mesos), Rancher, swarm。</li><li>尚未掌握對 Docker container 進行監控的技術，例如：Prometheus, cAdvisor。</li><li>業務量還不到需 scale 和使用負載平衡 (Load Balance) 的地步。</li></ol>要想好好發揮 Docker 原有的威力，勢必得架構做出大幅度的修正，但這些都不是單靠一位工程師就辦得到的，所以當時只有在整合 <a href="http://note.drx.tw/search/label/Jenkins" target="_blank">Jenkins CI</a> 的開發、測試的環境底下使用 Docker。<br /><br /><table align="center" cellpadding="0" cellspacing="0" class="tr-caption-container" style="margin-left: auto; margin-right: auto;
